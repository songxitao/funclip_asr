# grill-with-docs — 深度访谈 + 文档产出

## 何时用

- 有一个**还模糊的计划、设计或需求**，需要先和用户对齐理解
- 想**验证和打磨**某个方案，确保没有遗漏的决策分支
- 需要一个**共享的领域语言**（术语表），避免后续开发中用词混乱
- 需要记录**关键的、不可逆的架构决策**（ADR），让以后的人知道"为什么这么做"

用户输入 `/grill-with-docs` 来触发。如果只希望做访谈、不需要产出文档，用 `/grilling`。

## 核心工作流

本技能的精髓在于：**用一轮轮单问单答的"拷问"，沿着决策树逐级往下走，同时实时产出文档。**

### 流程概览

```
1. 启动 grilling 引擎（依次问问题，一次只问一个）
2. 每确认一个术语 → 立即写入 CONTEXT.md（领域术语表）
3. 每确认一个真正的架构决策 → 写入 docs/adr/ 下的 ADR
4. 完成时 → 用户有了 CONTEXT.md + 可能的 ADRs
```

## 完整指令

### 第一步：启动访谈（grilling 引擎）

对用户逐一向 "grilling" 模式提问，直到双方达成共享理解。规则如下：

- **一次只问一个问题**。问多个问题会让用户无所适从。
- **沿着决策树逐分支往下走**，优先解决依赖决策之间的顺序问题。
- **针对每个问题，给出你的推荐答案**，而不是丢个开放式问题让用户自己想。
- **能从代码库/环境中查到的事实，自己去查，不要问用户。** 但**决策**需要用户拍板。
- 在用户确认达成共享理解之前，**不要开始执行**任何操作。

### 第二步：实时维护 CONTEXT.md（领域术语表）

当访谈中一个术语被明确下来，**立即写入 CONTEXT.md**，不要批量堆积。

#### CONTEXT.md 格式

```markdown
# {Context Name}

{一两句话描述这个上下文是什么、为什么存在}

## Language

**{术语名}**:
{一两句话定义该术语是什么}
_Avoid_: {同义词，应避免使用的别名}

**{另一个术语}**:
{定义}
_Avoid_: {别名}
```

#### 规则

- **要有立场。** 当多个词指代同一概念时，选最好的，把其他列为 `_Avoid_`。
- **定义要精炼。** 一两句话，说清楚"是什么"，不说"做什么"。
- **只包含项目特有领域术语。** 泛编程概念（timeout、error type、utility pattern）不收录。
- **自然分组。** 如果术语自然分簇，在子标题下分组；都属同一领域就用平铺列表。

#### 多上下文仓库

- 仓库根有 `CONTEXT-MAP.md` → 读取它定位各个上下文
- 只有根 `CONTEXT.md` → 单上下文
- 都不存在 → 首次有术语确定时自动创建根 `CONTEXT.md`

### 第三步：审慎地记录 ADR

只有当以下**三个条件同时满足**时才提议创建 ADR：

1. **难逆转** — 将来改主意的成本很高
2. **不看上下文会令人困惑** — 将来的读者会想"为什么当初这么干？"
3. **真实权衡的结果** — 确实有多个备选方案，基于特定理由选了其中一个

缺任何一条，就跳过 ADR。

#### ADR 格式

文件放在 `docs/adr/`，顺序编号：`0001-slug.md`、`0002-slug.md`。

```markdown
# {简短决策标题}

{1-3 句话：上下文是什么，我们做了什么决定，为什么}
```

模板，仅此而已。一个 ADR 可以只有一个段落——记录"做了决定"和"为什么"本身就很有价值。

可选段落（只在确实有用时加）：
- **Status** 元数据（`proposed | accepted | deprecated | superseded by ADR-NNNN`）
- **Considered Options** — 被拒绝的替代方案值得记录时才写
- **Consequences** — 非显而易见的后续影响需要说明时才写

#### 适合 ADR 的情况

- 架构形态（"我们用 monorepo"、"写模型是 event-sourced，读模型投射到 Postgres"）
- 上下文之间的集成模式（"Ordering 和 Billing 通过领域事件通信，不是同步 HTTP"）
- 有锁定的技术选型（数据库、消息总线、认证服务、部署平台——不是每个库，是换掉要花一个季度的）
- 边界和范围决策（"Customer 数据由 Customer 上下文拥有；其他上下文只通过 ID 引用"）
- 偏离常规路径的故意选择（"我们手写 SQL 而非 ORM，因为 XXX"）
- 代码中看不到的约束（"因为合规要求不能用 AWS"）

### 第四步：对话中的持续精炼

在整个访谈中，保持以下行为：

- **对照已确认的术语挑战用户用词。** "你的术语表里 'cancellation' 定义为 X，但你刚才说的好像是 Y —— 究竟是哪个？"
- **精炼模糊语言。** 用户用模糊或过载的词时，提出精确的术语候选。"你说的 'account' —— 指的是 Customer 还是 User？这是两个不同的东西。"
- **用具体场景压力测试。** 当讨论领域关系时，编造能探测边界的边缘场景，迫使用户精确划定概念之间的界限。
- **与代码交叉验证。** 用户说某件事是如何工作的，检查代码是否一致。发现矛盾时提出来。
- **CONTEXT.md 保持纯术语表。** 禁止包含实现细节。不作为规格文档、草稿或决策仓库使用。

## 产出物

- **CONTEXT.md**（项目根目录或对应上下文目录）— 领域术语表
- **docs/adr/XXX-slug.md**（零到多个）— 架构决策记录

## 成功标准

- 一次只问一个问题，不 dump 问卷
- 术语一经确定立即写入 CONTEXT.md，用用户项目的语言
- 代码能回答的问题自己去查，不烦用户
- ADR 保持稀有 — 不会要求用户为可逆的选择盖章
